# Release v0.1.0

## What Happened

The first release candidate passed the repository's standard local verification but failed CI formatting. A narrow formatting-only repair was applied before the `v0.1.0` tag. The release created the first real downloadable artifacts, but the installer's successful prebuilt-download path was not smoke-tested against them.

## Root Cause

The standard verification command did not include every mandatory CI check: CI ran formatting and broader lint coverage that local verification omitted. Release evidence also stopped at the repair cell, so it did not durably record the tag commit, release workflow result, asset inventory, or installer outcome. Artifact production and consumption additionally share a string contract across the release workflow and installer.

## Recommendation

- Make the standard local verification command a superset of mandatory CI formatting, lint, test, and web checks.
- Tag only a commit already proven green by the full local suite and main-branch CI.
- After producing the first artifact under a new repository or artifact name, smoke-test the installer's real download, extraction, and execution path.
- Change release archive names and installer lookup/extraction paths atomically, and verify the contract mechanically.
- Record the tag SHA, workflow result, asset inventory, and installer smoke output in release evidence.

# Learning: Embedding the web UI + packaging parity with mdview

## Learning 1 — grep's exit code inverts pass/fail for negative-assertion verify commands

**Category:** failure
**Severity:** critical
**Tags:** [verify-commands, shell-scripting, cell-authoring]
**Applicable-when:** authoring or reviewing any bee cell whose `verify` command uses `grep`/`grep -c` to assert that a string is now ABSENT (a "no longer does X" truth).

### What Happened

`embed-pkg-4`'s original verify was `grep -c 'cp -r static' .github/workflows/release.yml`, meant to assert the string was gone. But `grep`'s own exit code is match-found (0) / not-found (1) regardless of `-c` — so this verify PASSED while the bug was still present and FAILED exactly when the fix was correctly applied. Caught during validating's PROOF SURFACE check, before any cell was dispatched. Fixed to `! grep -q 'cp -r static' .github/workflows/release.yml`.

### Root Cause

Author (planning-stage, this same session) used `grep -c` as if "successfully counted zero" meant the check passed, not realizing grep's exit status is independent of `-c` and still reports "did a match exist."

### Recommendation

When a cell's `must_haves.truths` contains a negative assertion ("no longer", "removed", "must not", "no longer copies"), any paired `verify` command containing a bare `grep`/`grep -c` (no leading `!`, no `-L`) must be rejected at cell-authoring or validation time — negate it explicitly. This bit this feature once already (embed-pkg-4); it will bite again in any future cell asserting absence via grep.

## Learning 2 — a feature that discovers an ordering constraint must check ALL its own cells against that same constraint, not just the obvious targets

**Category:** failure
**Severity:** critical
**Tags:** [ordering-bugs, self-consistency, verify-commands, install-scripts]
**Applicable-when:** planning or validating a feature that establishes a "X must happen before Y" rule anywhere in its own approach/plan.

### What Happened

Two separate instances of the exact same class of bug in one feature:
1. `install.sh` and `dev-deploy.sh` both ran `cargo build --release` before `npm run bundle` — harmless under the old runtime-disk-serving design, became a real bug once UI embedding went compile-time (would embed an empty/stale UI). Caught during planning discovery, fixed by reordering both scripts.
2. `embed-pkg-2`'s own `verify` command repeated the identical mistake: `cargo test --quiet && cargo clippy ... && (cd web && npm run bundle) && cargo test --quiet` — the first `cargo test` ran against an empty `static/`, aborting the `&&` chain before the bundle step ever ran. This was a BLOCKER caught only by the validating-phase persona panel, in the very same feature that had already identified "bundle must precede compile" as a systemic hazard for the shell scripts.

### Root Cause

The ordering constraint was correctly traced to install.sh/dev-deploy.sh (the "obvious" places a human would check) but not mechanically re-applied to the feature's own cell verify commands — a self-referential blind spot: the plan diagnosed the disease in others but not in itself.

### Recommendation

Once an approach/plan document states an ordering constraint ("X must precede Y"), grep every cell's `verify` string in that same feature for both tokens and flag any occurrence of Y appearing before X. This is currently a manual read (validating's PROOF SURFACE check caught it here) — worth a standing mechanical lint if this bee installation grows more compile-time-embedding-style features.

## Learning 3 — a `build.rs` guard decouples compile-time asset embedding from a generated, gitignored source directory

**Category:** pattern
**Severity:** standard
**Tags:** [rust-embed, build-rs, ci, gitignore]
**Applicable-when:** adding `#[derive(RustEmbed)] #[folder = "..."]` (or any proc-macro that reads a directory at compile time) where that directory is generated by a separate toolchain (here: `npm run bundle`) and is gitignored.

### What Happened

`static/` is gitignored (built by `npm run bundle`) and `ci.yml`'s `rust` job never runs `npm` — adding `rust-embed`'s folder macro without a guard would have broken every `cargo build`/`test`/`clippy` on a fresh clone and in that CI job. Fixed with a `build.rs` that `create_dir_all("static")` before the crate's own compilation, plus `cargo:rerun-if-changed=static` so a later real bundle triggers re-embedding.

### Root Cause

Compile-time embedding introduces a new hard dependency (the crate cannot compile without the folder existing) that a runtime-serving design never had — this is invisible until something makes the folder mandatory.

### Recommendation

Any time a Rust crate embeds a directory at compile time, and that directory's real content comes from a separate build step, add a `build.rs` guard that guarantees the directory exists (even empty) before compilation, independent of whether the other toolchain has run. Verify by simulating a fresh clone (`rm -rf <dir> && cargo test`) — this was the actual verification method used here (`embed-pkg-1`'s cap evidence), not just an assertion.

## Learning 4 — a validation persona panel caught 2 real BLOCKERs a single-pass validation missed

**Category:** pattern
**Severity:** standard
**Tags:** [validating, persona-panel, high-risk-lane]
**Applicable-when:** deciding whether a high-risk-lane feature's validating phase needs a multi-persona panel vs. a single plan-checker pass.

### What Happened

Three parallel background review agents (coherence+feasibility+security; product+scope-guardian; cold-pickup cell review) found: (1) install.sh's toolchain prerequisite check ran before the download attempt, silently defeating the download-first feature on exactly the machines it targeted; (2) `embed-pkg-2`'s verify command ordering bug (Learning 2). Both were structural BLOCKERs my own single-pass validation had not caught. 3 WARNINGs were also fixed (undocumented `debug-embed` choice, under-scoped doctor-output doc surfaces, missing curl download hygiene).

### Root Cause

A single validator (even a careful one) tends to check a plan's stated claims for internal consistency; a multi-persona panel with different lenses (security, product/UX, cold-pickup-implementability) independently re-derives the plan's implications from the raw evidence, catching gaps the plan's own narrative doesn't surface.

### Recommendation

For a high-risk lane feature spanning 3+ real domains (here: Rust + shell scripts + CI + docs), the persona-panel cost is worth it — this feature's 2 BLOCKERs would otherwise have shipped and only surfaced when a real user hit a toolchain-less machine or a CI run. `docs/history/embed-and-package-binary/reports/validation-current-slice.md` has the full finding-by-finding trail.

## Learning 5 — an implicit path coupling (config field doing double duty) is easy to miss reading code in isolation

**Category:** decision
**Severity:** standard
**Tags:** [config, data-paths, hidden-coupling]
**Applicable-when:** planning a change that makes a config field optional or removes its "always populated" guarantee.

### What Happened

`src/main.rs` derived the sqlite state-file path from `config.static_dir.parent()` — this only worked because `install.sh` happened to write `static_dir` as `$SHARE_DIR/static`, so its parent doubled as the app's data directory. Reading `main.rs` alone gives no hint this coupling exists; it was only found by tracing `install.sh`'s actual generated config during planning discovery. Fixed with an explicit `data_dir()` (confirmed byte-identical to the old implicit path by inspection, so no data migration needed for existing installs).

### Root Cause

A single config field silently served two purposes (asset location + data directory anchor) because one installer script happened to make them coincide — a convention that was never documented anywhere as a rule.

### Recommendation

Before removing a "must always be populated" guarantee from any config field, grep every use of that field across the codebase (not just its obvious/named purpose) — a `.parent()`/`.join()` derivation off a config path is a strong signal of a hidden second purpose worth tracing to its origin (here: the installer script that writes the config).
